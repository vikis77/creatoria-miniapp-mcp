# 05 · 结论与建议

> 调研日期：2026-05-31

## 1. 一句话结论

**项目用的 `miniprogram-automator@0.12.1` 已经是 npm 上的最新版（且 SDK 已停更两年半），最新微信开发者工具 v2.01.2510260 通过稳定的 CLI + WebSocket 协议仍兼容它——所以"更新代码适配新 CLI"是个伪命题，真正该做的是补 SDK 先天的连接/超时缺陷（已完成）。**

## 2. 关键发现矩阵

| # | 发现 | 证据强度 |
|---|------|---------|
| 1 | SDK npm latest = 0.12.1，无更高版本 | 强（npm registry 直查） |
| 2 | SDK 自 2023-11 停更，官方 changelog 停在 2022-09 的 0.12.0 | 强（官方文档） |
| 3 | 开发者工具活跃迭代，稳定版 v2.01.2510260（2025-10） | 强（官方下载页） |
| 4 | CLI `auto/--auto-port` 命令自 2021 工具 1.05+ 未变 | 强（官方 CLI 文档 + SDK 反编译比对） |
| 5 | SDK 依赖 ws@^6.1.3，无连接超时 → 印证卡死根因 | 强（本地依赖 + 源码） |
| 6 | 工具/SDK 通过 CLI 契约 + RPC 协议两层解耦，故老 SDK 驱动新工具 | 中（推断 + 实证） |

## 3. 行动建议（按优先级）

### ✅ 已完成（上一轮）
- connect/launch 外层 withTimeout + withRetry 兜底
- connect wsEndpoint 端口兜底 + 统一 127.0.0.1
- 修复 withTimeout 的 timer 泄漏
- 新增 connectTimeout 配置项
- 全量单测 501 passed

### 🟡 可选优化（非必须）
1. **package.json 钉精确版本**：`"miniprogram-automator": "0.12.1"`（去掉 `^`），防未来意外发布。
2. **文档注明工具版本基线**：在 README/runbook 注明"建议使用开发者工具稳定版，最低 1.05.2111232"。
3. **增加连接健康探测**：connect 前可先 TCP 探测端口是否开放（fail-fast），比等 30s 超时更友好。

### 🔵 长期关注（监控，不立即做）
- 关注微信是否发布 SDK 后继方案；当前无迹象。
- 若团队升级到 nightly 工具出现连接异常，优先怀疑协议变化，回退到稳定版验证。

## 4. 不建议做的事

- ❌ 不要试图"升级 SDK 到更新版本"——不存在更新版本。
- ❌ 不要为"适配新 CLI"重写启动逻辑——CLI 接口没变。
- ❌ 不要切换到 minium——架构不匹配（Python/真机），迁移成本远高于收益。
