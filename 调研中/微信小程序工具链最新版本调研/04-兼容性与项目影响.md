# 04 · 兼容性与项目影响

> 调研日期：2026-05-31

## 1. 核心兼容性判断

| 问题 | 结论 |
|------|------|
| 项目用的 SDK 是否最新？ | ✅ **是**。0.12.1 就是 npm latest，无更高版本可升 |
| 需要升级 SDK 吗？ | ❌ **无版本可升**。SDK 已停更，0.12.1 是终点 |
| 最新开发者工具（v2.01.2510260）会破坏当前 SDK 吗？ | ⚠️ **目前不会**。CLI `auto` 接口 + WebSocket 协议稳定，2021 至今未变 |
| CLI 路径/命令需要改吗？ | ❌ **不需要**。`launch.ts` 的 CLI 路径与 `auto/--auto-port` 参数与官方一致 |

## 2. 为什么"工具更新、SDK 停更"却仍能工作

工具与 SDK 通过**两层稳定契约**解耦：
1. **CLI 命令契约**：`cli auto --project --auto-port`（工具 1.05+ 起未变）
2. **WebSocket/RPC 协议契约**：自增 id 的 JSON 请求响应（多年未变）

只要这两层不变，工具内部怎么迭代都不影响 SDK。这是当前能"用老 SDK 驱动新工具"的根本。

## 3. 真正的风险（不是版本号，是脆弱性）

| 风险 | 等级 | 说明 |
|------|------|------|
| SDK 停更，未来工具若改协议则无人适配 | 🟡 中 | 目前无迹象，但失去官方维护意味着一旦破裂只能自行 patch |
| ws@6 老依赖 + 无连接超时 | 🔴 高（已修复） | 本项目已在 connect/launch 外包 withTimeout/withRetry 兜底 |
| jimp@0.6.4 老依赖 | 🟢 低 | 仅影响截图处理，功能正常 |
| Node 版本兼容 | 🟢 低 | SDK 纯 JS，无原生模块编译问题；项目要求 Node 18+ |

## 4. 对本项目两个任务的回答

### 任务1：代码是否需要更新以适配最新 CLI？
**不需要为"适配新 CLI"做任何改动**。理由：
- SDK 0.12.1 已是最新，且其 spawn 的 CLI 参数与 2025 最新工具的官方 CLI 文档完全一致
- CLI 路径 `/Applications/wechatwebdevtools.app/Contents/MacOS/cli` 未变
- `--auto-port`/`auto`/`--project` 命令未变

唯一"代码更新"是**健壮性兜底**（已在上一轮完成）：connect 端口兜底、connect/launch 外层超时重试、统一 127.0.0.1。这些与 CLI 版本无关，是补 SDK 的先天缺陷。

### 任务2：超时问题
根因与 SDK 版本无关（升级也解决不了，因为没有更高版本）。已通过本项目层兜底修复，详见 `docs/current/03-connect-screenshot-timeout-diagnosis.md`。

## 5. 是否要钉死/升级依赖

- **保持 `miniprogram-automator@0.12.1`**（已是最高），建议在 package.json 用精确版本而非 `^0.12.1`（防未来若有意外发布引入变化）—— 可选优化，非必须。
- **开发者工具**：建议团队统一使用稳定版（当前 v2.01.2510260），避免 nightly 引入协议变化导致连接异常。
